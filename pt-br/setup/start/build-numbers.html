No Android 8.0 (Oreo) e posteriores, as versões individuais são identificadas pelo formato PVBB.AAMMDD.bbb[.Cn] do código da versão, em que:

  • P representa a primeira letra do codinome da versão da plataforma (por exemplo, O de Oreo);
  • V representa um branch vertical compatível. Por convenção, P representa o branch principal da plataforma;
  • BB representa um código alfanumérico que permite ao Google identificar o branch exato do código em que a versão foi criada;
  • AAMMDD identifica a data do branch ou sincronização da versão com o branch de desenvolvimento. Não há como garantir que essa seja a data exata em que uma versão foi criada, já que é comum que pequenas variações adicionadas a uma versão existente reutilizem o mesmo código de data dessa versão;
  • bbb identifica versões individuais relacionadas ao mesmo código de data, iniciando sequencialmente com 001;
  • Cn é um alfanumérico opcional, que identifica um hotfix feito sobre uma versão PVBB.AAMMDD.bbb existente, começando com A1.

As versões mais antigas do Android usam um código de versão diferente e mais curto (por exemplo, FRF85B), em que:

  • A primeira letra é o codinome da família da versão, por exemplo, F refere-se a Froyo;
  • A segunda letra é um código de branch que permite ao Google identificar o branch exato do código em que a versão foi criada. Por convenção, R é o branch principal de versão;
  • A terceira letra e os dois dígitos seguintes são um código de data. A letra indica a contagem de trimestres (A = 1º trimestre de 2009, F = 2º trimestre de 2010 etc.). Os dois dígitos indicam a contagem dos dias dentro do trimestre (F85 se refere a 24 de junho de 2010). Não há como garantir que o código de data seja a data exata em que uma versão foi criada, já que é comum que pequenas variações adicionadas a uma versão existente reutilizem o mesmo código de data dessa versão;
  • A última letra identifica versões individuais relacionadas ao mesmo código de data, iniciando sequencialmente com A (o que é implícito e geralmente omitido por questões de brevidade).

Os branches froyo, gingerbread, ics-mr0, ics-mr1, jb-dev, jb-mr1-dev, jb-mr1.1-dev, jb-mr2-dev e kitkat-dev representam branches de desenvolvimento que não correspondem exatamente às configurações testadas pelo Google. Assim, esses branches podem conter uma variedade de alterações, além das versões oficiais marcadas, e podem não ter sido completamente testados.

Para diferenciar as versões, é possível visualizar uma lista de alterações associadas a cada projeto, executando o seguinte comando e transmitindo as duas tags de branch:

repo forall -pc 'git log --no-merges --oneline branch-1..branch-2'

Exemplo:

repo forall -pc 'git log --no-merges --oneline android-4.4.2_r2..android-4.4.2_r1'

Para enviar o resultado para um arquivo de texto:

  repo forall -pc 'git log --no-merges --oneline android-4.4.2_r2..android-4.4.2_r1' > /tmp/android-4.4.2_r2-android-4.4.2_r1-diff.txt

Módulos GPL do Honeycomb

Para o Honeycomb, todo o código-fonte da plataforma está indisponível. No entanto, as partes do Honeycomb licenciadas pela GPL e LGPL estão disponíveis nas seguintes tags:

Embora nenhum manifesto contenha exatamente esses módulos, alguns manifestos permitem a criação desses componentes. Os seguintes comandos funcionam para 3.0_r1.1:

repo init -b master -m base-for-3.0-gpl.xml
repo sync
repo forall -c git checkout android-3.0_r1.1

Para usar outra versão, troque o parâmetro git checkout e (se necessário) o parâmetro -m em repo init. O comando git checkout resulta em um erro para os projetos não GPL, em que não é possível encontrar a tag em questão.
